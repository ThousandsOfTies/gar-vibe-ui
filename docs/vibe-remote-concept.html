1. ビジョン

解きたい課題

AIエージェントは作業の途中で、人間の判断を必要とする。PCの前へ戻って状態を確認するだけの時間は、思ったより流れを止める。

本質的に必要なのは、状態を知ること、必要なら承認・却下すること、短い追加指示を返すこと。

体験の理想形

ソファや別作業中でも、VS Codeビューや手元の小さなデバイスで、エージェントが今どこにいるかを確認する。

画面には「作業中か」「待っているか」「どのエージェントが報告しているか」が短く出る。物理ボタンは次フェーズの入力導線になる。

UXは維持。 変わったのは実装方式です。VS Codeの他プラグインを外側から押すのではなく、対応エージェントがMCP/ブリッジへ状態を自己申告し、デバイスはその状態を購読します。

① Agent Cooperation

Codexや各社エージェントが、作業中・待機中・完了・失敗をMCPツールで明示的に報告する。

② Status Hub

Vibe Remoteが状態を保持し、VS Code内ビューや将来の物理デバイスへ配信する。

③ Physical UX

物理ボタンや小型画面はUXとして残す。現在はM5Stackドラフトで状態表示と簡易status送信を検証中。

3. 技術方針の変更

項目旧方式新方式
状態取得VS Code拡張が外側から推定MCP対応エージェントが自己申告
承認待ち他プラグインのUIやコマンドに依存エージェントが承認要求をブリッジへ出す
OK/NGVS Codeコマンド中継で押す将来のdecision responseとしてエージェントI/Fへ返す。現ドラフトではagentStatus送信を先に検証
対応範囲VS Code内部実装に左右されるMCP/ブリッジ対応したエージェントで安定
やらないこと： VS CodeのDOMや他拡張の非公開状態を無理に操作しない。これはプラグインごとに壊れやすく、汎用リモコンとして成立しにくい。

4. MCPステータスブリッジ

現行実装では、エージェントが次のMCPツールで状態を書き込めます。

Tool用途
vibe_remote_set_statusrunning, waiting, done, failed, idle を明示的に設定
vibe_remote_heartbeat作業継続中のハートビートを更新
vibe_remote_request_decision人間の判断待ちと選択肢を表示
vibe_remote_clear_status状態をidleへ戻す

6. デバイス連携

PC側設定

vibeRemote.bindAddress = 0.0.0.0
vibeRemote.discoveryEnabled = true
Vibe Remote: サーバを再起動
Vibe Remote: 接続トークンを表示

M5Stack側

cd vibe-remote/m5stack-client
pio run
pio run -t upload
pio device monitor

Wi-Fi SSID、パスワード、接続トークンは platformio.ini の build_flags で指定する。

現在地： M5Stackクライアントは接続検証用ドラフト。mDNS探索、WebSocket接続、hello/ping、BtnA/B/Cによるstatus送信まで実装済み。

8. ロードマップ

1
状態ブリッジを安定化 — MCPからのstatus/heartbeat/request_decisionをVS Codeビューへ表示。 実装済み
2
LAN discoveryを安定化 — mDNS広告とM5Stack側の自動発見を実機ネットワークで検証。 検証中
3
decision responseを設計 — OK/NGや選択肢をエージェントへ返す標準メッセージを追加。
4
入力UXを仕上げる — M5Stackのボタン、画面、トークン設定、再接続表示を実運用向けに整える。
