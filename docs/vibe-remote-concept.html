1. ビジョン

解きたい課題

AIエージェントは作業の途中で、人間の判断を必要とする。PCの前へ戻って確認するだけの時間は、思ったより流れを止める。

本質的に必要なのは、状態を知ること、承認すること、却下すること、短い追加指示を返すこと。

体験の理想形

ソファや別作業中でも、手元の小さなデバイスで OK / NG を返す。

画面には「何を待っているか」「どの選択肢があるか」「作業は進行中か」が簡潔に出る。

UXは維持。 変わったのは実装方式です。VS Codeの他プラグインを外側から押すのではなく、対応エージェントがMCP/ブリッジへ状態と承認要求を自己申告します。

2. システム全体像

① Agent Cooperation

Codexや各社エージェントが、作業中・待機中・完了・失敗をMCPツールで明示的に報告する。

② Status Hub

Vibe Remoteが状態を保持し、VS Code内ビューや将来の物理デバイスへ配信する。

③ Physical UX

物理ボタンや小型画面はUXとして残す。操作の宛先だけを、VS Code DOMではなくエージェントI/Fにする。

3. 技術方針の変更

項目旧方式新方式
状態取得VS Code拡張が外側から推定MCP対応エージェントが自己申告
承認待ち他プラグインのUIやコマンドに依存エージェントが承認要求をブリッジへ出す
OK/NGVS Codeコマンド中継で押す将来のdecision responseとしてエージェントI/Fへ返す
対応範囲VS Code内部実装に左右されるMCP/ブリッジ対応したエージェントで安定
やらないこと： VS CodeのDOMや他拡張の非公開状態を無理に操作しない。これはプラグインごとに壊れやすく、汎用リモコンとして成立しにくい。

4. MCPステータスブリッジ

現行実装では、エージェントが次のMCPツールで状態を書き込めます。

Tool用途
vibe_remote_set_statusrunning, waiting, done, failed, idle を明示的に設定
vibe_remote_heartbeat作業継続中のハートビートを更新
vibe_remote_request_decision人間の判断待ちと選択肢を表示
vibe_remote_clear_status状態をidleへ戻す
 
[mcp_servers.vibe_remote]
command = "/home/user/Yurufuwa/gar-vibe-ui/vibe-remote/scripts/node.sh"
args = ["/home/user/Yurufuwa/gar-vibe-ui/vibe-remote/scripts/mcp-server.js"]
env = { VIBE_REMOTE_TOKEN = "YOUR_TOKEN" }

緑 — done

赤 点滅 — failed / disconnected

7. ロードマップ

1
状態ブリッジを安定化 — MCPからのstatus/heartbeat/request_decisionをVS Codeビューへ表示。
2
decision responseを設計 — OK/NGや選択肢をエージェントへ返す標準メッセージを追加。
3
スタンドアロンUI検討 — VS Code内ビューに閉じず、スマホ/ブラウザ/物理デバイスから見えるHubにする。
4
物理デバイス実装 — M5StickC PLUS2/ESP32でstate streamを受け取り、ボタン入力をresponse bridgeへ返す。
